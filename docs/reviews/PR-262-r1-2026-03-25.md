## Verified correct

- **NULL coercion for source_notebook_id** (`R/db.R:1745`): `%||% NA_character_` correctly converts NULL to a length-1 NA string, which DuckDB stores as SQL NULL. Standard R idiom.
- **Length-safe fallbacks for is_overlap/community** (`R/db.R:1761-1762`): `rep(FALSE, nrow(nodes_df))` and `rep(NA_character_, nrow(nodes_df))` guarantee the fallback vector matches the length of all other columns. The `if/else` is explicit and avoids the `%||%` scalar trap.
- **Parameterized queries** (`R/db.R:1740-1745`): All SQL uses parameterized binding — no SQL injection risk.
- **Test coverage** (`tests/testthat/test-citation-network.R:226-293`): The new test exercises both NULL `source_notebook_id` and missing `is_overlap`/`community` columns simultaneously — exactly the production scenario that triggered the bug. Assertions verify node count, column values, and metadata round-trip.
- **Test isolation**: Each test creates its own in-memory DuckDB with `on.exit(dbDisconnect)` — no cross-test leakage.

## Pre-existing issues (filed)

1. **`paper_title %||% title` fallback can produce zero-length vector** (`R/db.R:1753`) -- Filed as #263. — If a `nodes_df` has neither `paper_title` nor `title` column, `nodes_df$paper_title %||% nodes_df$title` evaluates to `NULL`, and `as.character(NULL)` returns `character(0)`. This would cause the same `rapi_bind` error. Currently safe because all code paths provide one of these columns, but it's the same fragile `%||%` pattern this hotfix fixes for `is_overlap`/`community`.
