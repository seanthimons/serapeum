Create `R/mod_bulk_import.R` with UI and server functions.

**UI Function: `mod_bulk_import_ui(id)`**
Returns a hidden placeholder (the module uses modals for its workflow, triggered by the parent). Just return `tagList()` — all UI is in modals.

**Server Function: `mod_bulk_import_server(id, con, notebook_id, email_r, api_key_r, paper_refresh, db_path_r)`**
Parameters:
- `con` — reactive DuckDB connection
- `notebook_id` — reactive notebook ID string
- `email_r` — reactive OpenAlex email
- `api_key_r` — reactive API key
- `paper_refresh` — reactiveVal to trigger paper list refresh (increment to refresh)
- `db_path_r` — reactive DB file path for mirai worker

The server function implements this workflow:

**A. Import Modal (Step 1: Input)**
When parent calls `show_import_modal()` (exposed via returned list):

```r
showModal(modalDialog(
  title = tagList(icon("file-import"), "Bulk DOI Import"),
  tabsetPanel(id = ns("import_method"),
    tabPanel("Paste DOIs",
      textAreaInput(ns("doi_text"), "Paste DOIs (one per line or comma-separated)", rows = 8, placeholder = "10.1234/abc\nhttps://doi.org/10.5678/xyz\n..."),
    ),

    modalButton("Cancel"),

    actionButton(ns("start_import"), "Import", class = "btn-primary", icon = icon("file-import"), disabled = TRUE)
  ),
  size = "l",
  easyClose = FALSE
))
```

**B. Preview Logic**
`observeEvent(input$preview_btn)`:
1. Get DOI input from either textarea or file upload
2. For .bib files: call `extract_dois_from_bib(readLines(file$datapath))`, then feed extracted DOIs to `parse_doi_list()`
3. For CSV/txt files: call `parse_doi_list(readLines(file$datapath))`
4. For textarea: call `parse_doi_list(input$doi_text)`
5. Get existing notebook DOIs: `get_notebook_dois(con(), notebook_id())`
6. Separate valid DOIs into new vs duplicate
7. Store parsed results in reactiveValues for import step
8. Render preview panel:
   - "N valid new DOIs to import"
   - "N already in notebook (will be skipped)" — expandable
   - "N malformed (will be skipped)" — expandable with reasons
   - "N duplicates in input (deduplicated)" — if any
   - If .bib: "N entries without DOI field (skipped)"
   - Time estimate from `estimate_import_time(n_new_dois)`
   - Warning banner if 200+ DOIs: `div(class = "alert alert-warning", icon("triangle-exclamation"), "Large import: ", time_estimate)`
   - Warning for 500+ .bib entries similarly
9. Enable the Import button if there are new DOIs to fetch

**C. Import Execution (ExtendedTask pattern)**
Create ExtendedTask:
```r
import_task <- ExtendedTask$new(function(dois, notebook_id, email, api_key, db_path, run_name, interrupt_flag, progress_file) {
  mirai::mirai({
    source("R/bulk_import.R")
    source("R/db.R")
    source("R/utils_doi.R")
    source("R/api_openalex.R")
    source("R/interrupt.R")
    run_bulk_import(dois, notebook_id, email, api_key, db_path, run_name, interrupt_flag, progress_file)
  }, dois = dois, notebook_id = notebook_id, email = email, api_key = api_key,
     db_path = db_path, run_name = run_name, interrupt_flag = interrupt_flag,
     progress_file = progress_file)
})
```

`observeEvent(input$start_import)`:
1. Create interrupt flag and progress file
2. Get DOIs to import (new only, not duplicates)
3. Generate run_name: use input$run_name if provided, else "Import - {date}"
4. Record duplicate/malformed items in DB immediately (before API calls):
   - For each duplicate DOI: `create_import_run_item(con(), run_id, doi, "duplicate")`
   - For each malformed DOI: `create_import_run_item(con(), run_id, doi, "malformed", reason)`
   - But wait — the run_id is created inside the worker. Alternative: create the run in the main session, pass the run_id to the worker. Actually, per Research pitfall #2, the mirai worker needs its own DB connection. So:
   - Create the import_run in the main session BEFORE launching mirai
   - Record duplicate and malformed items in the main session
   - Pass run_id to the worker along with just the new DOIs
   - Worker only handles API fetching and recording success/not_found/api_error items
5. Close the input modal, show progress modal:
```r
removeModal()
showModal(modalDialog(
  title = tagList(icon("spinner", class = "fa-spin"), "Importing Papers"),
  tags$div(class = "progress", style = "height: 25px;",
    tags$div(id = ns("import_progress_bar"), class = "progress-bar progress-bar-striped progress-bar-animated",
             role = "progressbar", style = "width: 5%;", "5%")),
  tags$div(id = ns("import_progress_message"), class = "text-muted mt-2", "Initializing..."),
  footer = actionButton(ns("cancel_import"), "Cancel", class = "btn-warning", icon = icon("stop")),
  easyClose = FALSE
))
```
6. Invoke import_task with parameters
7. Start polling observer (invalidateLater(1000)) that reads `read_import_progress(progress_file)` and sends `session$sendCustomMessage("updateImportProgress", list(bar_id, msg_id, percent, message))`

**D. Cancel Handler**
`observeEvent(input$cancel_import)`: call `signal_interrupt(flag_file)`, update modal message to "Stopping..."

**E. Result Handler**
`observe({ result <- import_task$result(); req(result) })`:
1. Destroy progress poller
2. Close progress modal
3. Clean up interrupt/progress files
4. Update import run counts in main session DB
5. Refresh paper list: `paper_refresh(paper_refresh() + 1)`
6. Show results modal with categorized summary (see F)

**F. Results Summary Modal**
Show modal with:
- Header: "Import Complete" (or "Import Cancelled" if result$cancelled)
- Summary counts: "N imported, N failed, N duplicates skipped"
- Error categories as collapsible sections (use `<details><summary>` HTML):
  - "Malformed (N)" — expand to see DOI list
  - "Not found in OpenAlex (N)" — expand to see DOI list
  - "API error (N)" — expand to see DOI list
- "Retry Failed DOIs" button (only shown if failed_count > 0)
- Footer: "Close" button

**G. Retry Failed DOIs**
`observeEvent(input$retry_failed)`:
1. Get failed items from DB: `get_failed_import_items(con(), run_id)`
2. Extract DOIs, feed back into import pipeline (re-show progress modal)

**H. Import History Panel**
The server returns a function `render_import_history()` that the parent can call to get UI for the history section. OR better: the server renders into an output that the parent includes.

Add an output `output$import_history` that renders:
- Query `get_import_runs(con(), notebook_id())`
- For each run: collapsible card with run name, date, counts
- Delete button per run (calls `delete_import_run()`)
- Click to expand shows run items grouped by status

The parent module includes `uiOutput(ns("bulk_import-import_history"))` in the sidebar below the paper list.

```javascript
Shiny.addCustomMessageHandler('updateImportProgress', function(data) {
  var bar = document.getElementById(data.bar_id);
  var msg = document.getElementById(data.msg_id);
  if (bar) {
    bar.style.width = data.percent + '%';
    bar.setAttribute('aria-valuenow', data.percent);
    bar.textContent = data.percent + '%';
  }
  if (msg) {
    msg.textContent = data.message;
  }
});
```
