## Summary

Phase 52.1 fixes two UAT-blocking bugs in the search notebook's pagination system: (1) Refresh button showing triple toast on error (error toast + "No papers found" + "No new papers found"), and (2) Load More button silently failing to execute. The root causes are well-understood from code review during discuss-phase: Refresh's inner tryCatch returns a success-like value (`list(papers = list())`) instead of short-circuiting, allowing downstream toasts to fire; Load More's `req(pagination_state$has_more)` silently exits when the value is NULL (not FALSE).

The fix consolidates ~200 lines of duplicate code in `do_search_refresh()` and `do_load_more()` into a single unified `do_search()` function with mode parameter. This eliminates the split error handling that caused the cascade, provides a single toast policy, and prevents future drift.

**Primary recommendation:** Refactor into unified `do_search(mode)` function with mode-specific branching for cursor (NULL vs pagination_state$cursor), toast messages, and UI state (spinner swap for load_more only). Use proper control flow (early return() from function on error) and NULL-safe guards (isTRUE() instead of bare req()).

### Pattern 2: Early Return on Error (Short-Circuit Pattern)
**What:** Error handler calls return() to exit the entire function, preventing downstream code execution
**When to use:** When error state means no further processing should occur
**Example:**

# This code ONLY runs if no error occurred

  return()

### Pattern 3: NULL-Safe Guards with isTRUE()
**What:** Use isTRUE() instead of bare req() when distinguishing NULL from FALSE matters
**When to use:** When a reactive value might be NULL (uninitialized) vs FALSE (explicitly set)
**Example:**
```r
# Source: R documentation for isTRUE()
# BAD: req() silently exits on NULL — no feedback to user
req(pagination_state$has_more)  # Exits silently if NULL OR FALSE

# GOOD: isTRUE() only TRUE if value is TRUE (not NULL, not FALSE)
if (!isTRUE(pagination_state$has_more)) {
  showNotification("No more results available", type = "message")
  return()
}
```

### Anti-Patterns to Avoid
- **Returning "success-like" value from error handler:** `tryCatch(error = function(e) { ...; return(list(papers = list())) })` — This masks the error and allows downstream code to execute as if it succeeded
- **Using req() for NULL vs FALSE distinction:** req() silently exits on both NULL and FALSE — use isTRUE() when NULL means "not initialized" and FALSE means "explicitly disabled"
- **Nested if/else for error handling:** Creates deeply nested code and makes control flow hard to follow — use early return() pattern instead
- **Duplicate functions with minor differences:** Leads to drift over time as one gets fixed and the other doesn't — consolidate with mode parameter

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP retry logic | Custom retry loop with exponential backoff | httr2::req_retry() | Handles transient failures (429, 503), exponential backoff, max tries already implemented in build_openalex_request() |
| Error classification | String pattern matching per function | classify_api_error() helper | Centralized HTTP status → user message mapping prevents inconsistent error messages |
| Toast notifications | Custom HTML notification with details | show_error_toast() helper | Consistent expandable details UI, severity-based styling and duration |
| Abstract dedup | Client-side dedup with Set/Vector | DB query before insert | Handles race conditions, persists across sessions, simpler than maintaining client-side state |
| Icon state management | Manual updateActionButton calls scattered | on.exit() pattern | Guaranteed restoration even on error, single definition at function entry |

**Key insight:** Shiny reactive programming is inherently asynchronous. Error handling must account for partial execution — on.exit() ensures cleanup, early return() prevents cascades, and DB-based dedup handles concurrency better than client-side tracking.

## Common Pitfalls

### Pitfall 1: req() Silently Swallows NULL and FALSE
**What goes wrong:** `req(pagination_state$has_more)` exits silently if has_more is NULL (uninitialized) OR FALSE (no more results). User clicks Load More → nothing happens, no feedback.
**Why it happens:** req() is designed for reactive dependencies, not boolean guards. It treats NULL and FALSE identically (falsy).
**How to avoid:** Use isTRUE() when you need to distinguish NULL (not ready) from FALSE (explicitly disabled): `if (!isTRUE(pagination_state$has_more)) { showNotification(...); return() }`
**Warning signs:** Button click does nothing, no toast, no console error. Check for bare req() on values that might be NULL vs FALSE.

### Pitfall 2: Error Handler Returns "Success-Like" Value
**What goes wrong:** `tryCatch(error = function(e) { show_error_toast(...); return(list(papers = list())) })` — Error is caught and toast shown, but function continues executing as if API returned empty results. Downstream checks fire additional toasts ("No papers found", "No new papers found").
**Why it happens:** return() inside error handler only exits the handler, not the outer function. Execution resumes after tryCatch.
**How to avoid:** Use bare return() after error handling to exit the entire function: `tryCatch(error = function(e) { show_error_toast(...); return() })`. No value — just exit.
**Warning signs:** Multiple toasts firing on single operation. Error toast followed by "no results" toast. Trace execution flow — does error handler exit function or just handler?

### Pitfall 3: Code Duplication Leading to Asymmetric Fixes
**What goes wrong:** Refresh gets a bug fix (e.g., NULL-safe guard on filters), Load More doesn't. Or vice versa. Bugs fixed in one code path reappear in the other.
**Why it happens:** ~200 lines of nearly identical code in two functions. Developer fixes bug in one, forgets to apply to other. No enforcement mechanism.
**How to avoid:** Consolidate duplicate code into unified function with mode parameter. Single implementation → single fix location. Use tests to verify both modes behave identically for common cases.
**Warning signs:** Bug report says "Refresh works but Load More doesn't" or vice versa. Code review shows parallel implementations with slight differences.

### Pitfall 4: Forgetting to Restore UI State on Error
**What goes wrong:** Click Load More → API errors → spinner stays forever, button disabled, no way to retry. User must reload app.
**Why it happens:** updateActionButton() called before API, no corresponding restore call in error path.
**How to avoid:** Use on.exit() immediately after state modification. Runs regardless of success/error/return.
**Warning signs:** UI stuck in "processing" state after error. Button shows spinner indefinitely. Check for updateActionButton without corresponding on.exit.

### Pitfall 5: NULL-Unsafe if() Conditions
**What goes wrong:** `if (pagination_state$has_more) { ... }` throws error if has_more is NULL. Or worse, `if (filters$work_types) { ... }` on NULL value.
**Why it happens:** Reactive values and parsed JSON can be NULL. Bare if() expects TRUE/FALSE, not NULL.
**How to avoid:** Audit all if() conditions: `if (!is.null(x) && x) { ... }` or use isTRUE(x) for strict TRUE check. For req(), understand it exits on NULL — use isTRUE() if you need to distinguish.
**Warning signs:** Intermittent errors in logs like "argument is of length zero". Errors that only happen on first run (before state initialized). Check all if(reactive_value) and if(parsed$field) calls.

### Unified Search Function with Mode Branching
```r
# Source: Phase 52.1 CONTEXT.md (design pattern)
do_search <- function(mode = c("refresh", "load_more")) {
  mode <- match.arg(mode)

  nb_id <- notebook_id()
  req(nb_id)

  # Mode-specific guards
  if (mode == "load_more") {
    if (!isTRUE(pagination_state$has_more)) {
      showNotification("No more results available", type = "message")
      return()
    }
    req(!is_processing())
    is_processing(TRUE)
    updateActionButton(session, "load_more", icon = icon_spinner())
    on.exit({
      updateActionButton(session, "load_more", icon = icon_angles_down())
      is_processing(FALSE)
    })
  }

  # Common notebook retrieval
  nb <- get_notebook(con(), nb_id)
  req(nb$type == "search")
  if (mode == "refresh") {
    req(!is.null(nb$search_query) && nchar(nb$search_query) > 0)
  }

  cfg <- config()
  newly_added <- 0L

  # Mode-specific cursor and progress message
  cursor <- if (mode == "refresh") NULL else pagination_state$cursor
  progress_msg <- if (mode == "refresh") "Searching OpenAlex..." else "Loading more papers..."

  withProgress(message = progress_msg, value = 0, {
    # ... common API call, paper saving, dedup logic ...
  })

  # Mode-specific success toast
  if (mode == "refresh") {
    if (newly_added == 0L) {
      showNotification("No new papers found", type = "message")
    } else {
      showNotification(
        paste0("Added ", newly_added, " new paper", if (newly_added != 1L) "s" else "",
               " (", total_in_nb, " total in notebook)"),
        type = "message"
      )
    }
  } else {
    if (newly_added > 0L) {
      showNotification(
        paste0("Loaded ", newly_added, " more paper", if (newly_added != 1L) "s" else "",
               " (", total_in_nb, " total)"),
        type = "message"
      )
    } else {
      showNotification("No new papers (all duplicates)", type = "message")
    }
  }
}
```

## Open Questions

1. **Should Load More show a toast when clicked but has_more is NULL (not yet initialized)?**
   - What we know: Current code silently exits via req()
   - What's unclear: Is NULL a valid state (user clicked before first search completed), or should it be impossible (state initialized on search)?
   - Recommendation: Add explicit check with toast: "Please run a search first" if has_more is NULL. Treat NULL as uninitialized, FALSE as "no more results". Better UX than silent failure.

2. **Should error toast duration differ for warnings vs errors?**
   - What we know: show_error_toast() already implements this (5s for warnings, 8s for errors)
   - What's unclear: Is this duration sufficient for users to read technical details?
   - Recommendation: Keep existing durations. Expandable details section lets users re-read at their own pace.

3. **Should observeEvent handlers call unified function directly, or keep thin wrappers?**
   - What we know: Current pattern is `observeEvent(input$refresh_search, { do_search_refresh() })`
   - What's unclear: Should it become `observeEvent(input$refresh_search, { do_search("refresh") })` or keep wrapper?
   - Recommendation: Direct call with mode argument. Eliminates unnecessary indirection. Handler is already thin (just calls function).
