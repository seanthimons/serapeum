**Fix 1: Seed network SQL status mismatch (R/mod_bulk_import.R line 822)**

Change `i.status = 'imported'` to `i.status = 'success'`. The `create_import_run_item()` function in db.R stores status as `'success'` (not `'imported'`). This is a simple string replacement on one line.

**Fix 2: DOI fallback DB query format mismatch (R/mod_seed_discovery.R lines 75 and 138)**

The DB queries use `doi` directly, but in the manual lookup path (line 138), `doi` comes from `normalize_doi()` which returns URL-format (`https://doi.org/...`), while `abstracts.doi` stores bare format (`10.xxx/...`).

At line 75 (pre_fill path): wrap the doi parameter with `normalize_doi_bare()`:
```r
DBI::dbGetQuery(con(), "SELECT paper_id FROM abstracts WHERE doi = ? LIMIT 1", list(normalize_doi_bare(doi)))
```

At line 138 (manual lookup path): same fix:
```r
DBI::dbGetQuery(con(), "SELECT paper_id FROM abstracts WHERE doi = ? LIMIT 1", list(normalize_doi_bare(doi)))
```

This ensures bare-to-bare comparison regardless of whether the incoming DOI is URL-format or bare-format. `normalize_doi_bare()` already exists in `R/utils_doi.R` and handles both formats.

**Fix 3: Button vertical alignment (R/mod_citation_network.R lines 62-81)**

The button `div` wrappers use `d-flex align-items-center` but lack `h-100` to stretch to the full height of their `layout_columns` grid cell. Without `h-100`, the div is only as tall as the button itself, so `align-items-center` has no effect.

Change both button wrapper divs from:
```r
div(
  class = "d-flex align-items-center",
  actionButton(...)
)
```
to:
```r
div(
  class = "d-flex align-items-center h-100",
  actionButton(...)
)
```

This makes the div fill the full column height, and `align-items-center` then vertically centers the button relative to the taller slider/radio controls in adjacent columns.

The UAT reports a duplicate notification when using "Use as Seed", but the 40.1-02 plan already removed the showNotification from the pre_fill_doi observer (line 92 is now a comment). The user still sees a duplicate.

**Investigation approach:**

1. Check if `withProgress(message = "Looking up seed paper...", ...)` at line 72 is being perceived as a notification. `withProgress` shows a progress bar at the top of the page — this could look like a notification to the user.

2. Check if the pre_fill_doi observer triggers the lookup_btn observer. When `updateTextInput(session, "doi_input", value = doi)` is called at line 54, this changes the input value, but `observeEvent(input$lookup_btn, ...)` only fires on button click, not on input changes. So this is NOT the cause.

3. The most likely cause: `withProgress` at line 72 creates visual feedback that looks like a notification, followed by the actual "Paper found!" notification when the user manually clicks "Look Up" later. The user perceives the withProgress spinner + the notification as two notifications.

**Fix:** Remove the `withProgress` wrapper from the pre_fill_doi observer (lines 72 and 96). The paper preview appearing is sufficient feedback. Replace with a simple block (no progress indicator):

Change from:
```r
withProgress(message = "Looking up seed paper...", {
  # ... lookup code ...
})
```

To just the inner code without the `withProgress` wrapper. Keep all the lookup logic — just remove the progress indicator for the auto-triggered path. The auto-lookup is fast (single API call) and the paper preview appearing is the user's confirmation.

If after removing `withProgress` the user still reports a duplicate, the remaining notification is at line 164-168 (`showNotification("Paper found!", ...)`) which fires on manual lookup_btn click — that one is correct and should stay.
